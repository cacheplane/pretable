## Workflow you must follow

This repo uses the **superpowers** skill system. For each direction (B, C, D) you will:

1. Invoke **`superpowers:brainstorming`** to turn the rough direction into a design spec. Ask the user one question at a time. Propose 2–3 approaches with tradeoffs. Save the spec to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` and commit. Get user approval.

## Direction B — wire `@pretable/ui` into `apps/bench`

**Goal:** `apps/bench` uses the shared `<Nav>`, `<Footer>`, and design tokens from `@pretable/ui`. Visual identity matches `apps/playground` (once A lands). The adversarial grid comparison content stays; chrome around it changes.

**Scope:**

- Add `@pretable/ui` as a workspace dependency in `apps/bench/package.json`
- Import `@pretable/ui/tokens.css` and `@pretable/ui/components.css` at the app root (order matters: tokens before components)
- Replace any existing nav/header markup with `<Nav>` from `@pretable/ui`
- Replace any existing footer markup with `<Footer>` from `@pretable/ui`
- Remove or retire now-duplicate CSS in `apps/bench` that the design system supersedes (tokens, nav, footer — **not** the adversarial grid styles)
- Links in Nav: include at least "Playground", "Bench" (current, marked active), "Docs" (placeholder href if C hasn't landed)
- Footer: version from `package.json` (read via Vite's `import.meta.env` or a build-time constant — don't hardcode)
- CI status dot in Footer: initially hardcode `"green"` and TODO-comment a follow-up to wire it to a real source; don't overbuild
- Verify typecheck, lint, format, test, build all pass
- Visually verify the bench looks cohesive with `@pretable/ui` — launch `pnpm dev:bench` and inspect

**Out of scope for B:**

- Changing the adversarial grid content or bench logic
- New bench scenarios
- Cross-surface consistency with `apps/playground` (A handles that)

**Acceptance:**

- Bench renders with `<Nav>` and `<Footer>` styled identically to how `@pretable/ui` defines them
- No regressions to bench behavior (all benches still run, all scenarios still work)
- CI green

## Direction C — scaffold `apps/docs`

**Goal:** a new Mintlify-style docs surface that serves as the onramp. Sidebar + prose content area + right-rail TOC + ⌘K palette.

**Scope (minimum viable onramp):**

- New app at `apps/docs` — Vite + React 19.2 + TypeScript, same patterns as `apps/playground` and `apps/bench`
- Add to `pnpm-workspace.yaml` if a pattern doesn't already include it
- Add `dev:docs`, `build:docs` scripts at the repo root mirroring the other apps
- Uses `@pretable/ui` for Nav, Footer, tokens
- **Layout:** three columns on desktop — left sidebar (nav tree), center prose, right rail (page TOC built from `<h2>` / `<h3>`). Mobile: collapse sidebar to a drawer, hide TOC.
- **Content source:** MDX-backed pages (`@mdx-js/rollup` via Vite) stored in `apps/docs/content/` — start with **three pages only**: `index.mdx` (landing), `getting-started.mdx`, `concepts.mdx`. Populate with placeholder content the user can edit later; structure matters more than copy.
- **⌘K palette:** simple text-match over page titles and headings. Use `cmdk` if lightweight, otherwise build the 40-line version. Keyboard focus trap, Esc to close, Enter to navigate.
- **Routing:** client-side — use `react-router-dom` unless there's a stronger reason to pick another. Document the pick in the spec.
- Code blocks in MDX: use `<CodeBlock>` from `@pretable/ui`
- Callouts in MDX: use `<Callout>` from `@pretable/ui`

**Be disciplined:**

- No search backend, no versioning, no i18n, no analytics — YAGNI hard
- No new design tokens — if you need one, add it to `@pretable/ui` in a separate PR with user sign-off, don't inline hex in `apps/docs`
- Do **not** build a full docs generator; this is a Vite app that happens to render MDX

**Acceptance:**

- `pnpm dev:docs` boots and renders all three pages
- Sidebar navigation works, page TOC updates per-page, ⌘K finds headings
- CI green; no new lint/format/typecheck warnings in touched files
- Lighthouse-ish sanity check: no obvious a11y violations (landmarks present, heading order correct, skip-link, keyboard-navigable palette)
